1. ¿Qué hace este sistema?

Es una aplicación de escritorio que ayuda a administrar las prácticas profesionales de los estudiantes. Tres tipos de usuario entran con su número de personal y contraseña, y cada uno ve un menú distinto:

UsuarioQué puede hacer
Coordinador Registrar proyectos, asignar proyectos a estudiantes y programar las fechas de entrega.
Estudiante Llenar su autoevaluación.
Profesor Validar (aceptar o rechazar) los reportes mensuales.
Idea central El programa toma datos que el usuario escribe en pantalla, los revisa (validaciones) y los guarda en una base de datos MySQL. Eso es todo, repetido para 5 funciones.

2. Programación orientada a objetos en 5 minutos

La POO se basa en una sola idea: agrupar datos en "fichas" llamadas objetos.

Analogía: una tarjeta de cartón Imagina una tarjeta para anotar un proyecto, con huecos para el nombre, la fecha, etc. La plantilla vacía es la clase; cuando la llenas con datos reales, tienes un objeto.
// "class" = la plantilla (el molde de la tarjeta)
class Proyecto {
    String nombre;              // estos huecos se llaman ATRIBUTOS
    int    numeroMaximoParticipantes;
    LocalDate fechaInicio;
}

// crear un OBJETO = llenar UNA tarjeta nueva con datos
Proyecto inventario = new Proyecto(...);
PalabraQué esAnalogía
ClaseLa plantilla / el moldeLa tarjeta en blanco
ObjetoUna plantilla ya llenaUna tarjeta escrita
AtributoUn dato que guarda el objetoUn hueco de la tarjeta
MétodoUna acción que sabe hacerUna instrucción ("entrégate")

En este proyecto las "tarjetas" están en la carpeta modelo/pojo/: Proyecto, Estudiante, Usuario, ReporteMensual… Cada una guarda una cosa. A estas clases de puros datos se les dice POJO (objetos Java sencillos).

3. La arquitectura: 4 capas (MVC)

Para hacer un solo trámite colaboran 4 archivos, cada uno con un trabajo distinto. Es como una oficina con 4 departamentos:

1 · Vista (.fxml)

La cara: botones y campos que ves.

2 · Controlador

El cerebro: decide qué pasa al hacer clic.

3 · POJO

La ficha: guarda los datos de una cosa.

4 · DAO

El mensajero: habla con la base de datos.
   TÚ ──clic──▶  VISTA  ──▶  CONTROLADOR  ──▶  DAO  ──▶  BASE DE DATOS
                (.fxml)     (...Controller)   (...DAO)    (MySQL)
                 cara         cerebro         mensajero    archivo
¿Por qué separarlo así? Se llama MVC (Modelo–Vista–Controlador). Sirve para que cada cosa tenga su lugar: si cambias la pantalla no tocas la base de datos, y si cambias la base de datos no tocas la pantalla. Ordenado = fácil de mantener.
CapaCarpetaEjemplo
Vistasrc/sgpp/vista/FXML_RegistrarProyecto.fxml
Controladorsrc/sgpp/controlador/RegistrarProyectoController.java
POJO (datos)src/sgpp/modelo/pojo/Proyecto.java
DAO (acceso a datos)src/sgpp/modelo/dao/ProyectoDAO.java

4. El recorrido de un clic (ejemplo: Registrar Proyecto)

1 Llenas el formulario y pulsas Registrar (esto ocurre en la Vista).
2 El Controlador ejecuta el método clicRegistrar().
3 Pasa los datos por los filtros (validaciones). Si algo falla, avisa y se detiene.
4 Crea una ficha nueva: new Proyecto(...) (un POJO).
5 Se la entrega al mensajero: ProyectoDAO.registrar(proyecto).
6 El DAO la guarda en MySQL y la pantalla muestra "Registro Exitoso". ✅

Las otras 4 funciones siguen exactamente el mismo recorrido, cambiando solo el qué se valida y el qué se guarda.

6. Cómo funcionan las validaciones

Una validación es una pregunta: "¿se cumple esta condición? Si NO, no dejo continuar." En código eso es un if ("si...") seguido de return ("frena, no sigas").

Analogía: un portero El dato tiene que pasar por una fila de porteros (los if). Si uno solo lo rechaza, no entra y te avisa. Solo si pasa todos, la acción se realiza.
if ( algo está mal ) {
    avisar al usuario;
    return;        // detenerse aquí, NO guardar
}
// si el programa llega aquí, es que TODO estaba bien
guardar();

6.1 No permitir fechas anteriores a hoy (CU01 y CU11)

Se hace de dos maneras (cinturón y tirantes):

a) Apagar las fechas pasadas en el calendario para que ni se puedan elegir:

if (fecha.isBefore(LocalDate.now())) {  // si el día es ANTES que hoy
    setDisable(true);                   // lo pinta gris y lo bloquea
}

b) Revisar otra vez al pulsar Guardar, por si la escriben a mano:

if (fechaInicio.isBefore(LocalDate.now())) {
    mostrarAdvertencia("No se permite una fecha anterior a la actual.");
    return;
}

6.2 No permitir dos autoevaluaciones (CU15)

Antes de abrir la pantalla, el programa le pregunta al DAO si ya existe una:

if (AutoevaluacionDAO.existe(matricula)) {   // ¿ya hay una en la BD?
    mostrarAdvertencia("El expediente ya cuenta con una autoevaluacion.");
    return;                                // NO abre la pantalla
}

Y por dentro, existe() simplemente cuenta en la base de datos: si el resultado es mayor que 0, devuelve "verdadero".

6.3 La red de seguridad: try / catch

Hablar con la base de datos puede fallar (servidor apagado, etc.). Para que el programa no se rompa, se usa try ("intenta") y catch ("y si truena, haz esto"):

try {
    ProyectoDAO.registrar(proyecto);     // intenta guardar
} catch (SQLException e) {            // si falla la base de datos...
    mostrarError("No hay conexion con la base de datos.");
}
Patrón a recordar Validar → si está mal, return → si está bien, guardar dentro de try/catch. Ese mismo patrón se repite en los 5 casos de uso.

7. Base de datos, ConexionBD y DAO

Los datos viven en una base de datos MySQL llamada practicas_sgpp. El programa se conecta a ella con la clase ConexionBD.

Sin contraseñas en el código Los datos de conexión (usuario, contraseña, base) no están escritos en el código: se leen del archivo configuracion-bd.properties. Así se pueden cambiar sin recompilar y no quedan grabados dentro del programa.
// ConexionBD arma la dirección y entrega una conexión abierta
String url = "jdbc:mysql://" + ip + ":" + puerto + "/" + nombreBaseDeDatos;
return DriverManager.getConnection(url, usuario, password);

Cada DAO usa esa conexión para una tabla. Por ejemplo, así pide los estudiantes que aún no tienen proyecto, ya ordenados:

String consulta = "SELECT * FROM estudiante WHERE id_proyecto_asignado = 0 "
              + "ORDER BY semestre DESC, promedio DESC";

El DAO traduce cada fila de la tabla en un objeto POJO (una ficha Estudiante) y devuelve la lista al controlador. Ese es el puente entre la base de datos y los objetos.
